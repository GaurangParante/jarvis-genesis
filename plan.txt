What this project appears to be

This repo is building a personal AI assistant platform, not just a chatbot.
The direction is:
- A central Jarvis controller
- Multiple specialized agents
- Desktop automation and OS control
- Coding/project generation
- Knowledge and memory layers
- Future expansion to web, desktop app, and mobile app

In simple words, you are creating a modular "AI operating assistant" that can:
- Understand what a user wants
- Decide which specialist should handle it
- Execute actions on the computer
- Learn from past commands
- Eventually work across terminal, desktop app, web app, and mobile app

My read of the current intent

You are not building one single-purpose assistant.
You are building a system that can become the command layer for a user:
- Open apps
- Take screenshots / webcam photos / recordings
- Run coding tasks
- Research / content tasks
- Handle email, finance, fitness, social, and security workflows
- Possibly grow into a full personal AI product with a GUI and multi-device access

Core problem to solve

Right now the assistant is trying to do too many things directly from the core runtime.
To finish the project cleanly, the system should be split into:
- Brain layer: model/provider/router
- Skill layer: agents and tools
- Memory layer: cache, vector DB, logs, history
- Execution layer: safe command/action runner
- Interface layer: CLI, desktop app, web app, mobile app

Recommended target architecture

1. Core orchestration
- `Jarvis` should only coordinate the pipeline
- It should not contain provider-specific or platform-specific logic
- It should ask:
  - What does the user want?
  - Which agent should handle it?
  - What tools are allowed?
  - What is the final action plan?

2. Router layer
- Replace pure catch-word routing with a hybrid router:
  - rule-based fast path
  - embedding similarity path
  - LLM classification fallback
  - clarification fallback when confidence is low

3. Agent layer
- Each agent should have:
  - one responsibility
  - capabilities
  - tool access rules
  - examples
  - response schema

4. Tool layer
- Separate tools by domain:
  - desktop
  - files
  - browser
  - code
  - memory
  - content
  - system/security

5. Provider layer
- Use a provider interface so the system can switch between:
  - Groq
  - NVIDIA NIM
  - future providers later

6. Interface layer
- CLI first
- Then desktop app
- Then web app
- Then mobile app

How to change from Groq to NVIDIA NIM

Do not replace Groq in random files one by one.
Instead, create a model-provider abstraction.

Recommended steps:

1. Create a provider interface
- A single place for chat completion calls
- A single place for embeddings calls if needed
- A single place for error handling and retries

2. Move provider-specific code out of `LLMRouter`
- `LLMRouter` should ask a provider class for a response
- It should not know whether the backend is Groq or NIM

3. Create a `NIMProvider`
- It should map your existing chat-completion format to NIM's API format
- It should return the same normalized output shape used by the app

4. Keep a fallback provider
- If NIM fails or rate limits, optionally fall back to another provider

5. Update env config
- Replace or add env variables for NIM credentials and endpoint
- Keep secrets out of source code

6. Test JSON-only routing responses
- The LLM router relies on structured output
- You must validate that NIM returns consistent JSON
- Add repair logic for malformed JSON

7. If you use embeddings with NIM or another provider
- Put embedding generation behind the same abstraction
- Do not let semantic search depend on one vendor

How to stop "catch my word and decide the agent"

This part should be redesigned.

Current issue:
- The system is deciding mostly from keywords and static rules
- That is fast, but it is not intelligent or flexible
- It can misroute ambiguous requests

Better design:

1. Add an intent classifier
- The assistant should classify the request into a task type
- Examples:
  - code
  - desktop_action
  - research
  - content
  - email
  - finance
  - fitness
  - security
  - multi_step

2. Add a capability matcher
- Match intent against agent capabilities
- Select agents only if their capabilities fit the task

3. Add confidence scoring
- Every routing step should produce a score
- If confidence is low, ask a clarifying question

4. Allow multi-agent plans
- Some tasks should be split:
  - research first
  - then coding
  - then execution

5. Add user policy preferences
- Let the user set defaults like:
  - prefer one agent
  - prefer safe mode
  - require confirmation for system actions

6. Add a no-guess rule
- When confidence is too low, do not guess
- Ask one short clarifying question instead
